Permissioned Blockchain (Hyperledger Fabric) integration
=====================================================

Goal
----
This directory contains design notes and a starter Java chaincode module for integrating
AnnaSuraksha with a permissioned blockchain (Hyperledger Fabric). The intent is to
provide a production-grade, privacy-preserving ledger alternative to the current
in-database SHA256 chain used in the demo.

Why Fabric
----------
- Permissioned (consortium) model matches government deployments where multiple state
  or central agencies may run peers.
- Private Data Collections allow storing PII off-ledger while keeping hashes on the ledger.
- Pluggable consensus and ACLs give governance and performance control.

High-level design
-----------------
- Chaincode (this module) implements canonical operations:
  - CreateBeneficiary (store private data in a private collection, publish public digest)
  - QueryBeneficiary (read from private collection for authorized peers)
  - AnchorSnapshot (store Merkle root or snapshot metadata on public ledger)
  - QueryAnchor (fetch anchor record)
- Private data collections: `collectionBeneficiary` (permitting only authorized orgs to read PII).
- Endorsement policy: configurable per channel; for pilot use single-org endorsement.
- Identity: use Fabric CAs and MSPs for admins/peers; apps authenticate with client certificates.

Starter files
-------------
- fabric/chaincode : Java chaincode (Maven) skeleton using Fabric Java Contract API.

Deployment notes (pilot)
-----------------------
1. Install Hyperledger Fabric samples and prerequisites (docker, docker-compose, fabric-samples).
2. Use `fabric-samples/test-network` to create a channel and deploy the chaincode package.
3. Create a private data collection configuration for `collectionBeneficiary` and include in chaincode definition.
4. Use the AnnaSuraksha server to call chaincode via Fabric SDK (Java or Node) for operations.

Security & privacy
------------------
- Never write raw PII values to the channel's public state.
- Private data collections are stored off-chain on endorsing peers but hashed on the ledger.
- Use a backup and audit policy for private data; enable pvtdata dissemination controls.

Operator guide — quick start with fabric-samples test-network
----------------------------------------------------------
These steps show how to build the Java chaincode in this repo, deploy it to the `fabric-samples/test-network`,
and configure the AnnaSuraksha app to talk to the network using the Fabric Gateway Java SDK.

Prerequisites
 - Docker & Docker Compose
 - Hyperledger Fabric samples and binaries (https://github.com/hyperledger/fabric-samples)
 - Java 21 and Maven

High-level steps
1. Build the chaincode jar
   cd fabric/chaincode
   mvn -DskipTests clean package

2. Deploy to test-network (from fabric-samples/test-network)
   - From your fabric-samples directory run:
     ./test-network/network.sh deployCC -ccn beneficiary -ccp <path-to-repo>/fabric/chaincode -ccl java

   This will package and install the chaincode, approve and commit the definition, and instantiate it on the channel.

3. Create an application wallet and connection profile
   - Copy the org's MSP files to a wallet directory for the application. Example (Org1, test-network):

     mkdir -p wallet
     # copy cert
     cp fabric-samples/test-network/organizations/peerOrganizations/org1.example.com/users/Admin@org1.example.com/msp/signcerts/* wallet/admin-cert.pem
     # copy key (private key file name is unpredictable; use glob)
     cp fabric-samples/test-network/organizations/peerOrganizations/org1.example.com/users/Admin@org1.example.com/msp/keystore/* wallet/admin-key.pem

   - Use the sample connection profile generated by the test-network (organizations/peerOrganizations/org1.example.com/connection-org1.yaml)
     as your fabric.connectionProfile value.

4. Configure the AnnaSuraksha application
   In src/main/resources/application.properties or via environment variables set:

     fabric.enabled=true
     fabric.connectionProfile=/full/path/to/connection-org1.yaml
     fabric.walletPath=/full/path/to/wallet
     fabric.identity=Admin@org1.example.com
     fabric.channel=mychannel
     fabric.chaincode=beneficiary

5. Run the application and create a snapshot
   - Start the app. When MerkleService creates a snapshot it will attempt to call the chaincode anchorSnapshot
     function and persist the returned tx metadata on the Snapshot record.

Files included to help
- fabric/deploy-test-network.sh  — builds chaincode and invokes fabric-samples test-network deploy command (you pass your fabric-samples path)
- fabric/create-wallet.sh        — helper to copy MSP cert/key into a wallet directory for the app to use
- fabric/connection-profile-sample.yaml — minimal placeholder connection profile (edit to suit your test-network)

Security notes
- Use a dedicated, minimum-privilege identity for app interactions (do not use Org Admin in production).
- Protect the wallet directory (filesystem permissions) and never commit keys to the repo.
- Use TLS for gateway connections and ensure the connection profile points to secure endpoints.
